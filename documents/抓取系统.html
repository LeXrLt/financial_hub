信息抓取与 AI Agent 资料库系统

多源数据抓取 · 分表存储 · 给 AI Agent 提供稳定数据访问层

1系统目标

抓取多种来源的信息，保存到本地，供 AI Agent 后续读取、检索、总结、分析。

📰 公众号文章

图文内容，结构化字段 + 图片

▶️ YouTube 视频

视频 + transcript + 缩略图

🎙️ 小宇宙播客

长音频 + 转录文本

原则：来源数据结构不同，也不需要统一混合查询，每种来源 单独建表、单独爬取、单独管理。

3存储设计 · 双层分工

结构化数据进 Postgres，二进制文件进 NAS，Postgres 只存路径不存文件本体。

🗄️ Postgres（结构化）

结构化字段 正文文本 HTML 内容 transcript 原始链接 file local_path file metadata 抓取状态 转录状态 raw_data JSONB created_at updated_at
不存：图片、视频、音频文件本体

💾 NAS（文件本体）

公众号文章图片 YouTube 缩略图 YouTube 音频/视频 小宇宙封面图 小宇宙长音频 其他附件
Postgres 中只保存这些文件在 NAS 上的 路径。

4数据建模原则

每种来源单独建表，按各自真实结构建模，不强行统一字段。

❌ 避免的反模式

  • · 一张大表大量字段为空
  • · 不同来源字段互相干扰
  • · 主要数据全部塞进 JSONB
  • · 后期维护困难

✅ 推荐做法

  • · 每张主表保留 raw_data JSONB
  • · 保存原始 API 返回数据
  • · 方便后期补字段
  • · 方便调试和恢复数据

7爬虫逻辑

每个来源一个独立爬虫：wechat_crawler · youtube_crawler · xiaoyuzhou_crawler

每个爬虫的完整流程

1读取抓取目标
2抓取列表
3发现新内容
4抓取详情
5下载相关文件到 NAS
6如有需要，进行转录
7写入 Postgres
8更新抓取状态
第一版策略：抓取、下载、转录直接写在爬虫流程里。 后期数据量大、转录耗时长、失败重试复杂，再把下载和转录拆成独立任务。

8定时任务设计

第一版可以用简单 scheduler 或 cron 定时运行爬虫。

定时任务
→
运行对应 crawler
→
完整抓取流程
→
写入 Postgres + NAS

9状态设计

每条数据有简单状态，方便 Hub 和 Agent 判断数据是否可用。

通用状态

pending → processing → ready failed

小宇宙播客 · 转录状态（独立）

audio_downloaded → transcribing → transcript_ready transcript_failed
设计意图：转录状态独立，即使转录失败，也不会影响 metadata 和音频文件保存。

11Agent Skill 设计 · 数据访问层

AI Agent 不直接关心爬虫和数据库结构，只通过 Skill 读取数据。

主要能力

📰 公众号

  • 读取最新文章
  • 读取正文和图片

▶️ YouTube

  • 读取最新视频
  • 读取 transcript

🎙️ 小宇宙

  • 读取最新播客
  • 读取 transcript
  • 读取音频路径

通用查询能力

按时间范围读取
按关键词搜索
读取 metadata
读取 local_path

稳定输出格式

标题 来源 作者 / 频道 / 主播 发布时间 正文 / transcript 图片路径 音频路径 原始链接 metadata

不管底层来源结构如何变化，Skill 输出给 Agent 的字段保持稳定。
