
# 信息抓取与 AI Agent 资料库系统设计

## 1. 系统目标

系统用于抓取多种来源的信息，并保存到本地，供 AI Agent 后续读取、检索、总结和分析。

支持不同数据来源，比如

```text
公众号文章
YouTube 视频
小宇宙播客
```

这些来源的数据结构不同，也不需要统一混合查询，每种来源单独建表、单独爬取、单独管理。

## 2. 总体架构

```text
数据来源
  ↓
抓取目标列表
  ↓
定时爬虫
  ↓
Postgres + NAS
  ↓
Agent Skill
  ↓
AI Agent
```

各部分职责：

```text
Postgres：保存结构化数据、文本内容、transcript、文件 URL、local_path、metadata、raw_data
NAS：保存图片、视频、音频文件本体
Crawler：负责抓取、下载、必要时转录，并写入数据库
Hub：负责管理抓取目标和查看运行状态
Agent Skill：负责给 AI Agent 提供稳定的数据读取能力
```

## 3. 存储设计

### Postgres

Postgres 保存：

```text
结构化字段
正文文本
HTML 内容
transcript
原始链接
文件 local_path
文件 metadata
抓取状态
转录状态
raw_data JSONB
创建时间
更新时间
```

Postgres 不直接保存图片、视频、音频文件本体。

### NAS

NAS 保存：

```text
公众号文章图片
YouTube 缩略图
YouTube 音频 / 视频文件，如果需要
小宇宙封面图
小宇宙长音频
其他附件文件
```

Postgres 中只保存这些文件在 NAS 上的路径。

## 4. 数据建模原则

每种来源单独建表。

例如：

```text
wechat
youtube
xiaoyuzhou
```

每个来源按照自己的真实结构建模，不强行统一字段。

这样可以避免：

```text
一张大表里大量字段为空
不同来源字段互相干扰
主要数据全部塞进 JSONB
后期维护困难
```

但每张主表都建议保留：

```text
raw_data JSONB
```

用于保存原始 API 返回数据或原始抓取结果，方便后期补字段、调试和恢复数据。

## 5. 示例表结构

### 公众号文章

```text
wechat_accounts
wechat_articles
wechat_article_images
wechat_crawl_targets
```

### YouTube 视频

```text
youtube_channels
youtube_videos
youtube_video_transcripts
youtube_video_thumbnails
youtube_crawl_targets
```

### 小宇宙播客

```text
xiaoyuzhou_podcasts
xiaoyuzhou_episodes
xiaoyuzhou_episode_audio
xiaoyuzhou_episode_transcripts
xiaoyuzhou_crawl_targets
```

小宇宙播客是长音频内容，除了下载音频文件，还需要转录。  
音频文件保存在 NAS，转录文本保存在 Postgres。

## 6. 抓取目标设计

每种来源都有自己的抓取目标表。

例如：

```text
wechat_crawl_targets
youtube_crawl_targets
xiaoyuzhou_crawl_targets
```

抓取目标表用于记录：

```text
抓取对象
是否启用
抓取频率
上次抓取时间
上次抓取状态
失败原因
备注信息
```

例如：

```text
公众号：账号、文章入口、历史文章链接
YouTube：频道、播放列表、关键词
小宇宙：播客节目、主播、单集列表
```



## 7. 爬虫逻辑

每个来源一个独立爬虫：

```text
wechat_crawler
youtube_crawler
xiaoyuzhou_crawler
```

每个爬虫内部完成完整流程：

```text
读取抓取目标
抓取列表
发现新内容
抓取详情
下载相关文件到 NAS
如有需要，进行转录
写入 Postgres
更新抓取状态
```

抓取、下载、转录直接写在爬虫流程里。

后期如果数据量变大、转录耗时太长、失败重试变复杂，再把下载和转录拆成独立任务。

## 8. 定时任务设计

第一版可以用简单 scheduler 或 cron 定时运行爬虫。

```text
定时运行公众号爬虫
定时运行 YouTube 爬虫
定时运行小宇宙爬虫
```

整体逻辑：

```text
定时任务
  ↓
运行对应 crawler
  ↓
crawler 处理完整流程
  ↓
结果写入 Postgres 和 NAS
```

## 9. 状态设计

每条数据可以有简单状态，方便 Hub 和 Agent 判断数据是否可用。

通用状态：

```text
pending
processing
ready
failed
```

小宇宙播客可以额外有转录状态：

```text
audio_downloaded
transcribing
transcript_ready
transcript_failed
```

这样即使转录失败，也不会影响 metadata 和音频文件保存。

## 10. Hub 设计

Hub 是系统控制台，只需要做简单管理和监控。

主要功能：

```text
管理抓取目标
启用 / 停用抓取目标
查看最近抓取时间
查看最近抓取状态
查看失败原因
查看数据数量
查看播客转录状态
```

Hub 不负责复杂分析，只负责配置和查看系统是否正常运行。

## 11. Agent Skill 设计

Agent Skill 是给 AI Agent 使用的数据访问层。

AI Agent 不直接关心爬虫和数据库结构，只通过 Skill 读取数据。

主要能力：

```text
读取最新公众号文章
读取公众号文章正文和图片

读取最新 YouTube 视频
读取 YouTube transcript

读取最新小宇宙播客
读取小宇宙 transcript 和音频路径

按时间范围读取
按关键词搜索
读取 metadata
读取 local_path
```

Agent Skill 输出给 AI Agent 的内容尽量稳定：

```text
标题
来源
作者 / 频道 / 主播
发布时间
正文 / transcript
图片路径
音频路径
原始链接
metadata
```
